# Project Structure Guide

This document outlines the directory structure and content organization for The Lab Notebook.

## Directory Overview

```
the-lab-notebook/
├── components/           # React components
├── content/             # Main content directory
│   ├── docs-guides/     # Technical documentation and guides
│   ├── projects/        # Project portfolio and case studies  
│   ├── shortcuts.mdx    # Quick reference links
│   ├── docs.mdx         # Documentation index
│   └── _meta.js         # Navigation metadata
├── lib/                 # Utility libraries
├── public/              # Static assets
└── templates/           # Component usage examples
```

## Content Organization

### `/content/projects/`
**Purpose**: Project portfolio showcasing completed work and active development

**Contents**:
- `index.mdx` - Main project portfolio with ProjectCard and CompletedProjectCard components
- Individual project case studies (e.g., `e83-cas3-retrofit.mdx`)
- Project-specific subdirectories for multi-page documentation

**What belongs here**:
- Project overviews and case studies
- Implementation details and results
- Portfolio-style project presentations
- Before/after comparisons
- Technical achievements and outcomes

### `/content/docs-guides/`
**Purpose**: Technical documentation, guides, and reference materials

**Contents**:
- Multi-page guide subdirectories (e.g., `homelink-retrofit/`)
- Technical reference documentation
- Step-by-step tutorials and procedures
- Troubleshooting guides

**What belongs here**:
- How-to guides and tutorials
- Technical procedures and workflows
- Reference documentation
- Troubleshooting and diagnostics
- Code snippets and configurations
- Detailed technical breakdowns

### Root Content Files

#### `/content/shortcuts.mdx`
**Purpose**: Quick reference links and bookmarks

**Contents**:
- Development tools and utilities
- Automotive resources and databases
- Electronics components and suppliers
- Frequently accessed external links

#### `/content/docs.mdx`  
**Purpose**: Documentation index and technical references

**Contents**:
- Code libraries and snippets
- Circuit diagrams and schematics
- Technical specifications
- Quick reference tables

## Component Structure

### `/components/`
**Purpose**: Reusable React components

**Key Components**:
- `ProjectCard.jsx` - Active/planning project display
- `CompletedProjectCard.jsx` - Completed project showcase
- `LabNotebookHeader.jsx` - Site navigation
- `PhotoGrid.jsx` - Image gallery layouts
- `LightboxImage.jsx` - Expandable images

### `/templates/`
**Purpose**: Usage examples and documentation for components

**Contents**:
- `project-card-examples.mdx` - ProjectCard usage guide
- `completed-project-examples.mdx` - CompletedProjectCard usage guide

## Content Guidelines

### Projects vs Docs-Guides

**Use `/content/projects/` when**:
- Showcasing a completed project or active work
- Presenting results and outcomes
- Creating portfolio-style content
- Highlighting technical achievements

**Use `/content/docs-guides/` when**:
- Creating step-by-step instructions  
- Writing technical reference material
- Documenting procedures and workflows
- Providing troubleshooting information

### File Naming Conventions

- Use kebab-case for file and directory names
- Be descriptive but concise
- For multi-page content, create a subdirectory with `index.mdx`
- Use meaningful prefixes for related content (e.g., `e83-*` for E83-related projects)

### Navigation Structure

Navigation is controlled by `_meta.js` files in each directory:

```javascript
const meta = {
  'file-name': 'Display Title',
  'subdirectory': 'Section Title',
}
```

Order matters - items appear in the navigation in the order they're defined in the meta object.

## Asset Organization

### `/public/images/`
Organize images by content type:
- `/public/images/project-name/` - Project-specific images
- `/public/images/components/` - Component screenshots
- `/public/images/diagrams/` - Technical diagrams and schematics

### Image Optimization
- Use WebP format for web images when possible
- Provide both thumbnail and full-resolution versions
- Include descriptive alt text for accessibility
- Use consistent aspect ratios within galleries

## Best Practices

### Content Structure
- Start each page with a clear purpose statement
- Use consistent heading hierarchy (H1 for page title, H2 for major sections)
- Include relevant cross-references and links
- Provide context for technical content

### Component Usage
- Import components at the top of MDX files
- Use semantic HTML structure
- Follow established color schemes and styling patterns
- Test components across different screen sizes

### Documentation
- Keep this structure guide updated with changes
- Document any custom components in `/templates/`
- Use clear, descriptive commit messages
- Test all links and references before committing

This structure supports both portfolio presentation and technical documentation while maintaining clear separation of concerns and easy navigation.
